🔐 What is Authentication?
Understanding the foundation of digital identity and access control

Authentication is the process of verifying WHO you are before granting access to a system. Every time you log in to Gmail, Instagram, or your bank — the system is asking you to prove your identity. How securely you prove it determines how safe your account really is.

🧠
Factor 1 — Something You Know
Knowledge-based credentials that only you should know.
Examples: Password, PIN, Security question answer
📱
Factor 2 — Something You Have
A physical device or token that generates or receives a code.
Examples: Phone (OTP), Hardware security key, Smart card
👁️
Factor 3 — Something You Are
Biometric data unique to your physical body.
Examples: Fingerprint, Face ID, Iris scan, Voice recognition
🛡️ Single Factor vs Multi-Factor Authentication
Single Factor (Weak)
Only uses password. If stolen → instant account takeover. Used by 60% of people.
Multi-Factor (Strong)
Uses password + OTP or biometric. Blocks 99.9% of automated attacks even if password is stolen.

#1
Weak Password Policy
Critical ▼
Systems that allow weak, short, or commonly used passwords are extremely vulnerable to brute force attacks. During real-world testing using Hydra with the rockyou.txt wordlist (14 million passwords), the password "password" was cracked in under 3 seconds.
Most common passwords still used in 2026: 1. 123456 ← cracked in 0.001 seconds 2. password ← cracked in 0.001 seconds 3. qwerty ← cracked in 0.001 seconds 4. iloveyou ← cracked in 0.002 seconds 5. admin ← cracked in 0.001 seconds Strong password example: G#7kL$mP2@nQ9x!Wr ← takes 2 billion years to crack
⚠️ Impact: Attackers can crack weak passwords in seconds using automated tools. One compromised account can lead to full system breach, data theft, and financial loss.
✅ Fix: Enforce Strong Password PolicyMinimum 12 characters • At least 1 uppercase, 1 number, 1 symbol • Block commonly used passwords • Force password change every 90 days • Never allow password reuse
#2
Missing Multi-Factor Authentication (MFA)
Critical ▼
Without MFA, a stolen or guessed password gives an attacker immediate and complete access to the account. There is no second layer of verification to stop them. Microsoft research shows MFA blocks 99.9% of all automated attacks.
WITHOUT MFA: Attacker steals password → Full account access ❌ WITH MFA: Attacker steals password → System asks: "Enter OTP sent to your phone" → Attacker does not have your phone → ACCESS DENIED ✅ MFA reduces account compromise risk by 99.9%
⚠️ Impact: Every account without MFA is only one password breach away from full compromise. Password databases are sold on the dark web regularly.
✅ Fix: Implement Mandatory MFAUse TOTP apps (Google Authenticator, Authy) • SMS OTP as minimum • Hardware keys (YubiKey) for high-value accounts • Make MFA mandatory not optional for all users
#3
No Account Lockout Policy
Critical ▼
Without account lockout, an attacker can attempt unlimited password guesses using automated tools. This was demonstrated live during this internship — Hydra attempted 14 million passwords with zero resistance and found 16 valid passwords in just 3 seconds.
REAL TEST RESULT (Hydra brute force on DVWA): [DATA] attacking http-get-form://localhost:8080 [8080] host: localhost login: admin password: password [8080] host: localhost login: admin password: 123456 [8080] host: localhost login: admin password: iloveyou ...16 passwords found in 3 seconds. Time taken: 00:00:03 ← 3 SECONDS!
⚠️ Impact: Any account can be brute-forced given enough time. Without lockout, automated tools can try millions of passwords per minute with no resistance.
✅ Fix: Implement Account Lockout + Rate LimitingLock account after 5 failed attempts • Progressive delays between attempts • CAPTCHA after 3 failed attempts • Email alert on suspicious login activity • IP-based rate limiting
#4
Insecure Password Storage (Plain Text)
Critical ▼
Storing passwords as plain text means a single database breach exposes every single user's password immediately. Many major breaches (LinkedIn 2012, Adobe 2013, RockYou 2009) exposed millions of plain text or weakly hashed passwords.
❌ WRONG — Plain text storage: User: vennela | Password: MyPassword123 ❌ WRONG — Weak hashing (MD5): User: vennela | Hash: 9c42a1346e333a770904b2a2b37fa7d3 (crackable in seconds with rainbow tables) ✅ CORRECT — bcrypt with salt: User: vennela Hash: $2b$12$Kx8jH7mNpL3qR9sT2uVwOeYcZdJfIgHkMnBvCxAsDeFgHiJkLm (computationally impossible to reverse)
⚠️ Impact: If the database is breached, every user's password is instantly exposed. Users who reuse passwords across sites face complete identity compromise.
✅ Fix: Use Strong Password HashingUse bcrypt, Argon2, or scrypt algorithms • Always add a unique salt per user • Never use MD5 or SHA1 for passwords • Work factor of minimum 12 for bcrypt
#5
Weak Session Management
Critical ▼
After login, the server issues a session token to identify the user. If this token is predictable, long-lived, or transmitted insecurely, attackers can steal it and impersonate the user without ever knowing the password. This is called session hijacking.
❌ VULNERABLE session token: Token: user_12345_session ← predictable, guessable! Expiry: Never ← lives forever! Transmission: HTTP ← unencrypted! ✅ SECURE session token: Token: 8f3k9mN2pQ7rS4tU1vW6xY0zA5bC (cryptographically random, 256-bit) Expiry: 30 minutes of inactivity Transmission: HTTPS only HttpOnly + Secure cookie flags set
⚠️ Impact: Stolen session tokens give attackers full access to an authenticated account without needing credentials. XSS attacks (demonstrated in this internship) can steal these tokens.
✅ Fix: Secure Session ManagementUse cryptographically random 256-bit tokens • Expire after 15-30 mins inactivity • Set HttpOnly and Secure cookie flags • Invalidate on logout • Regenerate token after login
#6
Default Credentials Never Changed
High ▼
Millions of devices, applications, and systems ship with default credentials that administrators never change. Attackers maintain databases of all known default credentials and try them automatically. DVWA itself ships with admin/password as defaults.
Common default credentials attackers try first: admin / admin admin / password ← DVWA default (used in this internship) admin / 123456 root / root admin / (blank) user / user guest / guest administrator / administrator Shodan (a search engine for devices) indexes millions of internet-facing systems with default credentials still active.
⚠️ Impact: Attackers can gain instant access to routers, cameras, databases, and admin panels worldwide using default credentials. No hacking skills required.
✅ Fix: Enforce Credential Change on First LoginForce password reset on first use • Block default passwords in password policy • Audit all systems for default credentials quarterly • Use a password manager for complex unique passwords
#7
Missing HTTPS on Login Pages
High ▼
When a login form is served over HTTP instead of HTTPS, all data including usernames and passwords travels across the network as plain readable text. Anyone on the same WiFi network (coffee shop, airport, hotel) can intercept these credentials using tools like Wireshark.
HTTP (Unencrypted) — what anyone on your WiFi sees: POST /login HTTP/1.1 Host: example.com username=vennela&password=MyPassword123 ← FULLY READABLE by any network sniffer! ❌ HTTPS (Encrypted) — what anyone on your WiFi sees: TLSv1.3 Application Data 0x8F 0x3K 0x9M 0x2P 0x7R 0xS4 ... ← UNREADABLE encrypted gibberish ✅
⚠️ Impact: Credentials can be intercepted in real time on public WiFi networks. Man-in-the-Middle (MitM) attacks are trivial on HTTP sites.
✅ Fix: Enforce HTTPS EverywhereInstall SSL/TLS certificate (free via Let's Encrypt) • Redirect all HTTP to HTTPS automatically • Set HSTS header to force HTTPS • Use TLS 1.2 or higher only
#8
Weak Security Questions for Account Recovery
Medium ▼
Security questions like "What is your mother's maiden name?" or "What city were you born in?" can be answered by anyone who has spent 10 minutes looking at your social media profile. They create a backdoor into accounts that is often easier to exploit than the front door.
Common security questions vs social media reality: Q: "What is your mother's maiden name?" → Found on Facebook family tree in 2 minutes Q: "What city were you born in?" → Found on LinkedIn profile immediately Q: "What was your first pet's name?" → Found on Instagram "#throwback" posts Q: "What is your favourite sports team?" → Found on Twitter bio instantly
⚠️ Impact: Social engineering + social media research allows attackers to answer security questions and trigger unauthorized password resets, bypassing all other security measures.
✅ Fix: Replace Security Questions with Secure RecoveryUse email OTP for account recovery • Use authenticator app backup codes • Hardware security keys as recovery method • Remove security questions entirely from all systems
✅ Best Practices & Recommendations
Actionable security improvements every system should implement
🔑 Strong Password Policy
Enforce minimum 12 characters with uppercase, lowercase, numbers and symbols. Block the top 10,000 most common passwords. Force rotation every 90 days.
📱 Mandatory MFA
Require MFA for all user accounts without exception. Use authenticator apps over SMS where possible. Provide backup codes stored securely.
🔒 Account Lockout
Lock accounts after 5 consecutive failed attempts. Implement exponential backoff delays. Send email alerts on suspicious activity to the account owner.
🔐 Secure Password Storage
Hash all passwords with bcrypt or Argon2 with unique per-user salt. Never store plain text or use MD5/SHA1. Rehash on user login if algorithm is upgraded.
🌐 HTTPS Everywhere
Use TLS 1.2 or higher on all pages, not just login. Enable HSTS headers. Redirect all HTTP traffic to HTTPS automatically. Use free SSL from Let's Encrypt.
⏱️ Session Security
Expire sessions after 15-30 minutes of inactivity. Use cryptographically random tokens. Set HttpOnly and Secure cookie flags. Invalidate all sessions on password change.
🚫 No Default Credentials
Force credential change on first login for all systems. Audit quarterly for unchanged defaults. Use unique credentials per device and system. Maintain a credentials inventory.
📋 Regular Security Audits
Conduct penetration testing quarterly. Use automated vulnerability scanners monthly. Review authentication logs weekly for anomalies. Train all staff on authentication security annually.

📝 Conclusion
Summary and final recommendations from this security audit

🔐 Authentication Is Your First Line of Defence

This audit identified 8 authentication vulnerabilities across 3 risk levels — 5 critical, 2 high, and 1 medium. The most dangerous combination is weak passwords + no MFA + no lockout policy, which together allow automated attacks to compromise accounts in seconds, as demonstrated live using Hydra during this internship.

Implementing all 8 recommendations in this report would reduce the attack surface by an estimated 95%, making the system significantly more resilient against the most common real-world authentication attacks.
